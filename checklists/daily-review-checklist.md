## Purpose

This checklist structures the daily review of an `ops-log-monitor` report.
Complete it in order. Each step has a defined action and a place to record
your finding. A completed checklist serves as an auditable record that the
report was reviewed, not just generated.

Keep completed checklists with the report they reference, either in the
same directory or in your team's documentation system. Attach both the
checklist and the report to any helpdesk ticket raised as a result of
this review.

Estimated completion time for a NORMAL status report: 5–10 minutes.
Estimated completion time for a CRITICAL status report: 20–40 minutes
plus follow-up actions.

## Section 1: Pre-Review Verification

These steps confirm the report itself is trustworthy before you spend time
interpreting its content.

---

**Step 1.1 - Confirm the report was generated successfully**

Open the report file and locate the footer line:

```
Generated by [Windows/Linux] Operational Log Monitor v1.0.0
```

If this line is absent, the script may have crashed before writing the
report footer. Check the script execution log before proceeding.

- [ ] Footer present - report completed normally
- [ ] Footer absent - check execution log before continuing

**Execution log location:** _________________________________

---

**Step 1.2 - Confirm the monitoring window is correct**

The report header shows:

```
Monitoring Window: YYYY-MM-DD HH:MM:SS to YYYY-MM-DD HH:MM:SS
```

Verify this window is contiguous with yesterday's report window. A gap
between yesterday's end time and today's start time represents a period
with no coverage.

- [ ] Window is contiguous with previous report - no gap
- [ ] Gap detected - investigate why (missed scheduled run, reboot
  during scheduled time, etc.)

**Previous report end time:** _________________________________

**Gap (if any):** _________________________________

---

**Step 1.3 - Check the Execution Notes section**

Scroll to the bottom of the report and review the Execution Notes section
if present. This section lists non-fatal issues that occurred during report
generation (modules that returned errors, log size warnings, permission
degradation, etc.).

- [ ] No Execution Notes present - clean run
- [ ] Execution Notes present - reviewed and recorded below

**Notes summary:** _________________________________

**Step 1.4 - Verify system information is plausible (if included)**

An unexpected last boot time (system rebooted without a corresponding
maintenance window) is itself a finding requiring investigation.

- [ ] Last boot time matches known maintenance history
- [ ] Unexpected reboot detected - flagged for investigation

## Section 4: WARN Finding Review

WARN findings do not require same-day action but should be reviewed for
patterns before signing off. Answer the questions below rather than
reviewing each WARN event individually.

---

**Step 4.1 - Are any WARN findings recurring from previous reports?**

Compare today's WARN findings against this week's preceding reports. A
finding that appears as WARN every day without reaching CRIT thresholds
may indicate a slow-developing problem or a misconfigured system that
warrants investigation even without a CRIT escalation.

- [ ] No recurring WARN patterns identified
- [ ] Recurring WARN pattern identified

**Recurring pattern description (if any):**

_________________________________
_________________________________

**Action taken for recurring WARN:**

- [ ] Investigating - ticket: _____________
- [ ] Known expected activity - adding to exception list

---

**Step 4.2 - Do any WARN findings appear correlated with CRIT findings?**

A WARN finding that occurs at a similar time to a CRIT finding in a
different category may indicate the two are related (e.g., a WARN for
a service account logon failure immediately preceding a CRIT for a
lockout, or a WARN for an unexpected sudo command immediately preceding
a CRIT for a sudoers modification).

- [ ] No correlation identified
- [ ] Potential correlation identified

the next run to prevent the same false positive from appearing again and
degrading the signal-to-noise ratio.

**Step 5.2 - Is the next scheduled run configured correctly?**

Occasionally verify that the scheduled task (Windows) or cron/systemd
timer (Linux) is still active and not in a failed state. This is not
required at every daily review - monthly is sufficient unless a missed
run was detected in Step 1.2.

## Quick Reference - Common Findings and Initial Response

| Finding Type | First Action |
|---|---|
| Account lockout | Contact account owner directly (not by email). Determine whether the user initiated the activity. |
| After-hours logon following failures | Contact account owner directly. If not confirmed by owner: disable account immediately. |
| Service crash (security-relevant) | Check whether service is currently running. Check for related events within 30 minutes either side. |
| SeDebugPrivilege use by unexpected process | Treat as probable credential-dumping attempt. Escalate before completing investigation. |
| Sudoers file modification | Compare against change management system immediately. Do not defer. |
| SELinux AVC denial burst | Check `restorecon` need on affected path. If source process is unexpected: escalate. |
| OOM killer invocation | Check current memory utilization. Schedule memory capacity review. |
| Unexpected system shutdown | Cross-reference with facilities/power incident log. Check hardware health. |
