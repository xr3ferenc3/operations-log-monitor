# Linux Log Sources Reference

## Purpose of This Document

This document explains every log source queried by `ops-log-monitor` on
Linux systems. For each source it covers:

- What the log source is and what generates it
- Where it is located on disk
- How the framework queries it
- What information it contains
- What prerequisites are required for it to be populated
- How to verify it is working correctly
- Common reasons it may be empty or missing

This document is written for administrators who want to understand where
the data in their Linux reports comes from, and for those troubleshooting
missing or incomplete report sections.

---

## Log Architecture Overview

RHEL 9 uses two parallel logging systems that serve different purposes and
must both be understood to interpret log monitor output correctly.

### systemd-journald

journald is the primary logging subsystem on RHEL 9. It collects log
entries from:

- The kernel (via /dev/kmsg)
- systemd and systemd units (service start, stop, failure events)
- Applications that log via the syslog protocol or directly to journald
- Standard output and standard error of systemd-managed services

journald stores log data in a structured binary format in
/var/log/journal/ (persistent) or /run/log/journal/ (volatile, lost at
reboot). The binary format is not human-readable directly but is queried
efficiently using journalctl.

Persistent journal storage requires /var/log/journal/ to exist:

```bash
# Verify persistent journal storage
ls -ld /var/log/journal/

# If the directory does not exist, create it and restart journald
mkdir -p /var/log/journal
systemd-tmpfiles --create --prefix /var/log/journal
systemctl restart systemd-journald
```

On a fresh RHEL 9 installation, persistent journal storage is enabled by
default. On systems upgraded from earlier RHEL versions, it may need to be
enabled manually.

### auditd

auditd is the Linux kernel audit subsystem daemon. It operates at a
different layer than journald. While journald collects messages from
userspace processes and the kernel's logging interface, auditd receives
structured audit records directly from the kernel's audit framework.

auditd is the authoritative source for:

- System call auditing
- File and directory access auditing
- User authentication events (PAM)
- sudo and su executions
- SELinux access control decisions

auditd writes to /var/log/audit/audit.log in a structured key-value text
format. This file is the most forensically reliable log source on a Linux
system because it is generated by the kernel audit subsystem and is harder
to manipulate from userspace than syslog-based logs.

auditd is installed and enabled by default on RHEL 9:

```bash
# Verify auditd status
systemctl status auditd

# Verify audit log is being written
ls -lh /var/log/audit/audit.log
```

### Relationship between journald and auditd

journald and auditd are not redundant. They capture overlapping but
distinct information. For example:

- A failed SSH login attempt generates records in both journald (from the
  sshd process logging via syslog) and auditd (from the PAM subsystem
  generating USER_AUTH audit records)
- A sudo execution generates a record in journald (from the sudo process)
  and a more detailed record in auditd (from the kernel audit subsystem
  capturing the execve system call if auditing rules are configured)

The framework queries both systems and reconciles the data where overlap
exists to avoid duplicate reporting.

## Log Sources by Module

### Authentication Module (`auth-events.sh`)

#### journald - sshd messages

**What it contains:**
SSH daemon log messages covering connection attempts, authentication method
used, authentication success or failure, and session lifecycle events.

**Query method:**

```bash
journalctl --no-pager --since="YYYY-MM-DD HH:MM:SS" \
           --until="YYYY-MM-DD HH:MM:SS" \
           --identifier=sshd \
           --output=short-iso
```

**Key message patterns extracted:**

| Pattern | Meaning |
|---|---|
| `Failed password for` | Password authentication failure for existing account |
| `Failed password for invalid user` | Authentication attempt for nonexistent account |
| `Invalid user` | Connection attempt using a username not present on the system |
| `Accepted password for` | Successful password authentication (uncommon if keys enforced) |
| `Accepted publickey for` | Successful key-based authentication |
| `Connection closed by authenticating user` | Client disconnected during authentication (common in scans) |
| `Too many authentication failures` | Client reached MaxAuthTries limit |
| `Did not receive identification string` | Port scanner or misconfigured client |

**Prerequisites:**
sshd must be running and must log to journald (default on RHEL 9).

**Verify:**

```bash
journalctl --identifier=sshd --since="1 hour ago" | head -20
```

---

#### journald - PAM messages

**What it contains:**
Pluggable Authentication Module messages covering authentication events
for all PAM-aware services: SSH, su, sudo, local console login, and
any application using PAM.

**Query method:**

```bash
journalctl --no-pager --since="YYYY-MM-DD HH:MM:SS" \
           --until="YYYY-MM-DD HH:MM:SS" \
           --identifier=sshd \
           --identifier=su \
           --identifier=sudo \
           --identifier=login \
           --output=short-iso \
           | grep -E "pam_unix|pam_faillock|authentication failure"
```

**Key message patterns extracted:**

| Pattern | Meaning |
|---|---|
| `pam_unix.*authentication failure` | Generic PAM authentication failure |
| `pam_faillock.*user .* locked out` | Account locked by pam_faillock after repeated failures |
| `pam_unix.*session opened` | Session started for authenticated user |
| `pam_unix.*session closed` | Session ended |

**Note on pam_faillock vs pam_tally2:**
RHEL 9 uses pam_faillock for account lockout management. Earlier RHEL
versions used pam_tally2. The *RHCSA Red Hat Enterprise Linux 9
Certification Study Guide* (Jang) covers pam_faillock configuration.
If pam_tally2 messages appear, the system may be running an older
PAM configuration that was not updated during an OS upgrade.

---

#### /var/log/secure (fallback)

**What it contains:**
On RHEL 9, /var/log/secure is populated by rsyslog receiving auth and
authpriv facility messages from journald forwarding. It contains the same
authentication messages as journald but in traditional syslog text format.

**When the framework uses it:**
The framework queries journald as the primary source. /var/log/secure is
used as a fallback if journald returns no results, which can occur if
persistent journal storage is not configured or if journald was restarted
without persistent storage, losing in-memory entries.

**Verify:**

```bash
ls -lh /var/log/secure
tail -20 /var/log/secure
```

**Note:**
/var/log/secure may not exist on minimal RHEL 9 installations where
rsyslog is not installed. In that case, journald is the only source.
The framework handles the absence of /var/log/secure gracefully.

### Service Module (`service-events.sh`)

#### journald - systemd unit state changes

**What it contains:**
systemd logs unit state transitions, start and stop events, and failure
events directly to journald. These records are generated by systemd itself,
not by the services, so they are present even if a service crashes without
logging anything.

**Query method:**

```bash
# Units that entered failed state
journalctl --no-pager --since="YYYY-MM-DD HH:MM:SS" \
           --until="YYYY-MM-DD HH:MM:SS" \
           --identifier=systemd \
           --output=short-iso \
           | grep -E "Failed to start|entered failed state|failed with result"
```

**Key message patterns extracted:**

| Pattern | Meaning |
|---|---|
| `Failed to start` | Unit failed during start attempt |
| `entered failed state` | Unit is now in failed state |
| `failed with result 'exit-code'` | Unit process exited with non-zero status |
| `failed with result 'signal'` | Unit process was killed by a signal (crash or OOM) |
| `failed with result 'timeout'` | Unit did not start within its configured timeout |
| `Stopped` (preceded by no deliberate stop) | Unexpected unit stop |

**Prerequisites:**
systemd must be the init system (guaranteed on RHEL 9). Journal must be
accessible.

**Verify:**

```bash
journalctl --identifier=systemd --since="1 hour ago" \
  | grep -E "failed|Failed" | head -20
```

---

#### systemctl - failed units enumeration

**What it contains:**
A direct query of current systemd unit state, not a log query. Used to
capture units currently in failed state at report generation time.

**Query method:**

```bash
systemctl list-units --state=failed --no-legend --no-pager
```

**How the framework uses it:**
The framework captures the list of currently failed units at report time
and compares it against the units that appeared in the journal as having
entered failed state during the monitoring window. Units currently failed
that do not appear in the journal window are flagged as possibly having
failed before the monitoring window began (persistent failures).

### Privilege Module (`privilege-events.sh`)

#### /var/log/audit/audit.log - sudo executions

**What it contains:**
auditd generates USER_CMD records for every sudo command execution when
auditd rules for sudo are configured. These records include the username,
the terminal, the working directory, and the exact command executed.

**Record format example:**

```
type=USER_CMD msg=audit(1719830400.123:456): pid=12345 uid=1001 \
auid=1001 ses=5 subj=unconfined_u:unconfined_r:unconfined_t:s0-s0:c0.c1023 \
msg='cwd="/home/jsmith" cmd=737973746374 ...' exe="/usr/bin/sudo" \
hostname=server01 addr=192.168.1.10 terminal=pts/0 res=success
```

**Note on command encoding:**
The cmd field in auditd records is hex-encoded to handle special characters
and spaces in command arguments. The framework decodes these values before
including them in the report. The raw hex is not presented to the
administrator.

**Query method:**

```bash
ausearch --start "MM/DD/YYYY HH:MM:SS" \
         --end "MM/DD/YYYY HH:MM:SS" \
         --message USER_CMD \
         --raw \
  | aureport --input - --comm --summary 2>/dev/null
```

Or directly:

```bash
grep "type=USER_CMD" /var/log/audit/audit.log \
  | awk -v start="$start_epoch" -v end="$end_epoch" \
    'BEGIN{...} ...'
```

**Prerequisites:**
auditd must be running. The audit rules for sudo must be loaded
(see threat-model.md for required rules). /var/log/audit/audit.log
must be readable by the user running the script (requires root or
membership in the adm group with appropriate ACLs).

**Verify:**

```bash
# Run a sudo command and check for the record
sudo ls /root
grep "USER_CMD" /var/log/audit/audit.log | tail -5
```

---

#### /var/log/audit/audit.log - su executions

**What it contains:**
USER_AUTH and USER_ACCT records for su authentication attempts, and
USER_CMD style records for su sessions when auditd rules for su are
configured.

**Key record types:**

| Type | Meaning |
|---|---|
| USER_AUTH | Authentication attempt (success or failure) |
| USER_ACCT | Account validation (the account being switched to exists and is permitted) |
| CRED_ACQ | Credentials acquired (su succeeded) |
| CRED_DISP | Credentials disposed (su session ended) |

**Verify:**

```bash
su --command "whoami" - root 2>/dev/null || true
grep "type=USER_AUTH" /var/log/audit/audit.log | tail -5
```

#### journald - sudo fallback

**What it contains:**
sudo logs its executions to syslog/journald even without auditd rules.
The journald records are less detailed than auditd records (no hex-encoded
command arguments, no working directory) but are sufficient to determine
who ran sudo and what command was run at a high level.

**Query method:**

```bash
journalctl --no-pager --since="YYYY-MM-DD HH:MM:SS" \
           --until="YYYY-MM-DD HH:MM:SS" \
           --identifier=sudo \
           --output=short-iso
```

**Key message patterns:**

| Pattern | Meaning |
|---|---|
| `USER ; TTY=` | Successful sudo execution (includes user and command) |
| `user NOT in sudoers` | Failed sudo attempt by unauthorized user |
| `incorrect password attempts` | Failed sudo due to wrong password |
| `command not allowed` | User is in sudoers but not permitted to run this command |

**When the framework uses it:**
Primary source when auditd is not running. Secondary confirmation source
when auditd is running, to cross-reference and catch any records that
auditd may have dropped (rare but possible under high load).

### System Health Module (`system-health-events.sh`)

#### journald - kernel storage messages

**What it contains:**
The Linux kernel logs storage-related errors directly to the kernel message
ring buffer, which journald captures via /dev/kmsg. These messages cover
SCSI/SATA errors, I/O errors, filesystem errors, and storage controller
events.

**Query method:**

```bash
journalctl --no-pager --since="YYYY-MM-DD HH:MM:SS" \
           --until="YYYY-MM-DD HH:MM:SS" \
           --dmesg \
           --output=short-iso \
           | grep -iE "I/O error|SCSI error|blk_update_request|EXT4-fs error| \
             XFS.*error|Buffer I/O error|end_request|ata.*error|nvme.*error"
```

**Key message patterns:**

| Pattern | Meaning |
|---|---|
| `Buffer I/O error on dev` | Read or write I/O error on a block device |
| `blk_update_request: I/O error` | Block layer I/O error (device-level) |
| `EXT4-fs error` | EXT4 filesystem-level error |
| `XFS (sda): Internal error` | XFS filesystem error |
| `ata1.00: error` | ATA/SATA device error |
| `SCSI error: return code` | SCSI command error |
| `nvme nvme0: I/O` | NVMe device error |

**Why this matters operationally:**
Storage errors in kernel logs frequently precede disk failure by days to
weeks. Catching these early allows disk replacement during a planned
maintenance window rather than during an emergency outage.

#### journald - OOM killer messages

**What it contains:**
When the Linux kernel's Out of Memory killer terminates a process to
reclaim memory, it logs the event to the kernel message ring buffer with
details about which process was killed, its memory usage, and the system
memory state at the time.

**Query method:**

```bash
journalctl --no-pager --since="YYYY-MM-DD HH:MM:SS" \
           --until="YYYY-MM-DD HH:MM:SS" \
           --dmesg \
           --output=short-iso \
           | grep -E "Out of memory|oom_kill_process|Killed process|oom-kill"
```

**Key message patterns:**

| Pattern | Meaning |
|---|---|
| `Out of memory: Killed process` | OOM killer terminated a process |
| `oom_kill_process` | OOM killer function called |
| `oom-kill:constraint=CONSTRAINT_NONE` | System-wide OOM (not a cgroup limit) |
| `oom-kill:constraint=CONSTRAINT_MEMCG` | Container or cgroup memory limit hit |

**How the framework uses it:**
Every OOM kill event is reported as CRIT. OOM kills indicate the system
does not have sufficient memory for its current workload and may cause
data loss if the killed process was performing I/O.

#### journald - unexpected shutdown indicators

**What it contains:**
When a system reboots cleanly, systemd logs the shutdown sequence to
the journal. When a system reboots uncleanly (power loss, kernel panic,
hard reset), the next boot's journal begins without a preceding clean
shutdown message. The framework detects this gap.

**Detection method:**
The framework checks the journal for the previous boot using:

```bash
journalctl --list-boots --no-pager
```

This lists all recorded boots with their start and end times. A boot
entry with no end time (or an end time that matches the start time of
the next boot without a clean shutdown message in between) indicates
an unclean shutdown.

**Verify:**

```bash
# List all recorded boots
journalctl --list-boots --no-pager

# Check for clean shutdown messages in previous boot
journalctl --boot=-1 --no-pager \
  | grep -E "Reached target.*Shutdown|Stopped.*System"
```

### Kernel Module (`kernel-events.sh`)

#### /var/log/audit/audit.log - SELinux AVC denials

**What it contains:**
Every time SELinux denies an access attempt, the kernel audit subsystem
generates an AVC (Access Vector Cache) denial record in the audit log.
The record identifies the source context (the process attempting access),
the target context (the resource being accessed), the permission class,
and the specific permission that was denied.

**Record format example:**

```
type=AVC msg=audit(1719830400.456:789): avc:  denied  { read } for  \
pid=2345 comm="httpd" name="secret.conf" dev="sda1" ino=131073 \
scontext=system_u:system_r:httpd_t:s0 \
tcontext=unconfined_u:object_r:admin_home_t:s0 tclass=file permissive=0
```

**Fields the framework extracts:**

| Field | Meaning |
|---|---|
| comm | Process name that triggered the denial |
| scontext | SELinux context of the denying process |
| tcontext | SELinux context of the accessed resource |
| tclass | Object class (file, dir, tcp_socket, etc.) |
| denied permission | The specific permission denied (read, write, execute, etc.) |
| permissive | 0 = enforcing (action blocked), 1 = permissive (action allowed but logged) |

**Query method:**

```bash
grep "type=AVC" /var/log/audit/audit.log \
  | awk -v start="$start_epoch" -v end="$end_epoch" \
    'BEGIN{...} function parse_epoch(ts){...} ...'
```

Or using ausearch:

```bash
ausearch --start "MM/DD/YYYY HH:MM:SS" \
         --end "MM/DD/YYYY HH:MM:SS" \
         --message AVC \
         --raw
```

**Prerequisites:**
SELinux must be in enforcing or permissive mode. Disabled SELinux generates
no AVC records. auditd must be running to write AVC records to
/var/log/audit/audit.log. If auditd is not running, AVC records go to
/var/log/messages or journald instead, with reduced detail.

**Verify:**

```bash
# Check SELinux mode
getenforce

# Check for recent AVC denials
ausearch -m AVC --start recent 2>/dev/null | head -30
```

**Interpreting AVC denials:**
Not all AVC denials indicate an attack or a problem. Common benign sources:

- Application installed without correct SELinux file contexts - run
  `restorecon -Rv /path/to/application` to fix
- Application updated and the policy has not been updated - may require
  a policy module update
- Custom application that has never had a policy written for it

Suspicious sources:
- A confined process (httpd_t, mysqld_t) attempting to access files
  outside its expected context - possible exploitation attempt
- A process running as unconfined_t generating denials - unusual, as
  unconfined processes are rarely denied
- Denials for network access from a process that should not make network
  connections

*RHCSA Red Hat Enterprise Linux 9 Certification Study Guide* (Jang)
covers SELinux context management and AVC denial interpretation in detail.
The audit log format used in this document follows current RHEL 9
auditd behavior.

#### journald - kernel warnings and errors

**What it contains:**
Kernel messages logged at warning level (kern.warning) and above,
captured by journald from /dev/kmsg.

**Query method:**

```bash
journalctl --no-pager --since="YYYY-MM-DD HH:MM:SS" \
           --until="YYYY-MM-DD HH:MM:SS" \
           --dmesg \
           --priority=warning \
           --output=short-iso
```

**Key indicators extracted:**

| Pattern | Meaning |
|---|---|
| `Oops:` | Kernel oops (recoverable kernel bug, system may be unstable) |
| `BUG:` | Kernel bug assertion - serious instability |
| `WARNING:` | Kernel warning - may or may not indicate a problem |
| `Call Trace:` | Stack trace following an oops or bug - used for debugging |
| `general protection fault` | CPU detected invalid memory access |
| `kernel panic` | Unrecoverable kernel error (system will halt or reboot) |

**How the framework uses it:**
Kernel oops and BUG events are reported as CRIT. Kernel WARNING events
are reported as WARN. The framework does not report all kernel warnings -
many are expected and benign on specific hardware configurations. The
filtering patterns in the configuration file can be tuned to suppress
known-benign warnings for a specific environment.

## Log Retention and Monitoring Window Limitations

The monitoring window a report can cover is bounded by how far back each
log source retains data.

### journald retention

journald retention is governed by /etc/systemd/journald.conf. Key parameters:

| Parameter | Default on RHEL 9 | Meaning |
|---|---|---|
| SystemMaxUse | 10% of filesystem | Maximum total size of persistent journal |
| SystemKeepFree | 15% of filesystem | Space journald will leave free |
| MaxRetentionSec | 0 (unlimited) | Maximum age of journal entries |
| MaxFileSec | 1month | How often journal files are rotated |

On a system with a small /var partition and high log volume, the journal
may only retain a few days of data. On a system with generous /var space,
the journal may retain months.

To check current journal disk usage and estimated retention:

```bash
journalctl --disk-usage
journalctl --verify
```

### auditd retention

auditd retention is governed by /etc/audit/auditd.conf. Key parameters:

| Parameter | Default | Meaning |
|---|---|---|
| max_log_file | 8 (MB) | Maximum size of each audit log file |
| num_logs | 5 | Number of rotated audit log files to keep |
| max_log_file_action | ROTATE | What to do when max size is reached |

The default configuration retains approximately 40 MB of audit logs
(8 MB × 5 files). On a busy system, this may represent only hours of
data. Production systems requiring longer audit retention should increase
num_logs or configure auditd to forward to a remote log server.

To check audit log retention:

```bash
ls -lh /var/log/audit/
aureport --start "30 days ago" --end now --summary 2>/dev/null \
  || echo "Audit data does not extend 30 days"
```
